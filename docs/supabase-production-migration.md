# Supabase Production Migration Guide

## Overview

This document describes how to transfer the TelentFest system from staging to a new production Supabase project.

## Prerequisites

- A new Supabase project (Pro plan or above for 500+ concurrent users)
- Access to project settings → API → Project URL, anon key, service_role key
- Supabase CLI installed (`npm install -g supabase`)

---

## Step 1: Create Production Project

1. Go to [supabase.com/dashboard](https://supabase.com/dashboard)
2. Create a new project
3. Note the project URL and service_role key

## Step 3: Create Storage Buckets

### Bucket: `participant-photos`

- **Public bucket** required for PDF rendering (the PDF generator loads images client-side via URL)
- **Privacy note**: Since this is public, use random UUID filenames (no personal info in paths)
- **Alternative**: For stricter privacy, use private bucket with signed URLs and update the PDF generator to fetch via server proxy

### SQL to create bucket:

```sql
INSERT INTO storage.buckets (id, name, public)
VALUES ('participant-photos', 'participant-photos', true)
ON CONFLICT (id) DO NOTHING;
```

### Storage Policies (applied by migration file, verify manually):

```sql
-- Public read
CREATE POLICY "Public read participant-photos"
ON storage.objects FOR SELECT
USING (bucket_id = 'participant-photos');

-- Service role full access
CREATE POLICY "Service role full access participant-photos"
ON storage.objects FOR ALL
TO service_role
USING (bucket_id = 'participant-photos')
WITH CHECK (bucket_id = 'participant-photos');

-- Admin management
CREATE POLICY "Admins manage participant-photos"
ON storage.objects FOR ALL
TO authenticated
USING (bucket_id = 'participant-photos' AND public.is_admin())
WITH CHECK (bucket_id = 'participant-photos' AND public.is_admin());
```

### Bucket: `event-images` (if not already created)

```sql
INSERT INTO storage.buckets (id, name, public)
VALUES ('event-images', 'event-images', true)
ON CONFLICT (id) DO NOTHING;
```

## Step 4: Set Environment Variables

| Variable | Source | Example |
|----------|--------|---------|
| `VITE_SUPABASE_URL` | Supabase project → Settings → API → Project URL | `https://xxxxx.supabase.co` |
| `VITE_SUPABASE_ANON_KEY` | Supabase project → Settings → API → anon public key | `eyJhbGciOi...` |
| `SUPABASE_URL` | Same as VITE_SUPABASE_URL | `https://xxxxx.supabase.co` |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase project → Settings → API → service_role key | `eyJhbGciOi...` |
| `VITE_PUBLIC_SITE_URL` | Your deployed site URL | `https://telentfest.com` |

**Never commit `SUPABASE_SERVICE_ROLE_KEY` to version control.**

## Step 5: Seed Data

Run seed SQL for:

1. **Events** — Create your events with proper dates, venues, capacities
2. **Seat sections** — Already generated by migration for active events
3. **Seats** — Already generated by migration for active events
4. **Admin users** — Insert into `user_roles` table:

```sql
INSERT INTO public.user_roles (user_id, role)
VALUES ('<auth-user-id>', 'admin')
ON CONFLICT DO NOTHING;
```

**Do not copy:**
- Test payments (payments table)
- Staging registrations
- Staging passes
- Staging photos
- Staging audit data

## Step 6: RLS Verification

After migration, verify RLS is enabled:

```sql
SELECT tablename, rowsecurity FROM pg_tables
WHERE schemaname = 'public'
AND tablename IN ('event_seat_sections', 'event_seats', 'seat_holds',
  'seat_bookings', 'seat_allocation_audit', 'rate_limits');
```

All should show `rowsecurity = true`.

## Step 8: Rollback Plan

### Rollback a migration:

```sql
-- Drop new tables:
DROP TABLE IF EXISTS public.rate_limits CASCADE;
DROP TABLE IF EXISTS public.seat_allocation_audit CASCADE;
DROP TABLE IF EXISTS public.seat_bookings CASCADE;
DROP TABLE IF EXISTS public.seat_holds CASCADE;
DROP TABLE IF EXISTS public.event_seats CASCADE;
DROP TABLE IF EXISTS public.event_seat_sections CASCADE;

-- Drop new columns:
ALTER TABLE public.registrations DROP COLUMN IF EXISTS seat_allocation_status;
ALTER TABLE public.registrations DROP COLUMN IF EXISTS photo_storage_path;
ALTER TABLE public.registrations DROP COLUMN IF EXISTS photo_url;
ALTER TABLE public.registrations DROP COLUMN IF EXISTS photo_uploaded_at;
ALTER TABLE public.registrations DROP COLUMN IF EXISTS photo_mime_type;
ALTER TABLE public.registrations DROP COLUMN IF EXISTS photo_size_bytes;
ALTER TABLE public.registrations DROP COLUMN IF EXISTS event_name_snapshot;
ALTER TABLE public.registrations DROP COLUMN IF EXISTS event_city_snapshot;
ALTER TABLE public.registrations DROP COLUMN IF EXISTS event_date_snapshot;
ALTER TABLE public.registrations DROP COLUMN IF EXISTS event_start_time_snapshot;
ALTER TABLE public.registrations DROP COLUMN IF EXISTS event_end_time_snapshot;
ALTER TABLE public.registrations DROP COLUMN IF EXISTS event_venue_snapshot;
ALTER TABLE public.passes DROP COLUMN IF EXISTS seat_section_name;

ALTER TABLE public.passes DROP COLUMN IF EXISTS seat_number;

ALTER TABLE public.passes DROP COLUMN IF EXISTS event_id;

-- Drop functions:

DROP FUNCTION IF EXISTS public.generate_event_seats;
DROP FUNCTION IF EXISTS public.increment_rate_limit;

```

## Known Limitations

1. **Photo storage is public bucket** — Required for client-side PDF rendering. Mitigation: random UUID filenames, no PII in paths.
2. **Rate limiting uses database table** — Works in staging but for production at scale, consider Redis or edge middleware.
3. **No email/SMS/push notifications** — Database schema ready but delivery not implemented.
4. **Admin pagination** — Passes list loads all rows. For 10,000+ passes, add cursor-based pagination.
5. **Complex seat map** — Auto-allocates deterministically; interactive seat picker not built.
